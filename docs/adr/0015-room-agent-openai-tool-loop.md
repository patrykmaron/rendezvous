# 0015. OpenAI Responses tool loop orchestrates the room agent inside Trigger.dev

**Status:** Accepted
**Date:** 2026-07-21

## Context

[0011](0011-trigger-dev-orchestration.md) sketched the analysis pipeline as a fixed chain of Trigger.dev tasks run directly, with no LLM in the loop. But the product is a group chatting together beside a shared map, and the driving product rule is that the agent's *answers are visual*: the map carries the plan, chat is at most a short caption, never a table of scores. That rule needs something that can hold a conversation, decide when it has enough to act (origins present? routing succeeded?), and narrate ŌĆö while never being trusted with the numbers themselves, since an LLM restating a fairness score is a hallucination risk the group would have no way to catch. The group also needs to *see* the agent working live ŌĆö status ("scanning candidate areasŌĆ”", "computing 40 journeysŌĆ”") and its reply streaming in ŌĆö without polling Postgres, and the realtime channel that carries this must not leak one room's agent activity to another room's members.

## Decision

`room-agent` (`packages/tasks/src/trigger/room-agent.ts`) is a Trigger.dev v4 `schemaTask`, triggered from `apps/web/lib/agent-trigger.ts`'s `startAnalysis` (called from the header button and from an `@agent` chat mention) and tagged `room:<roomId>`. It runs an **OpenAI Responses API** tool loop ŌĆö `openai.responses.create`, streamed, `openai` SDK v6 ŌĆö against `gpt-5.6-terra` (`packages/tasks/src/trigger/agent/model.ts`; this model family is served only by Responses, Chat Completions is legacy/unsupported for it). The toolset is fixed and small: `generate_candidates`, `compute_route_matrix`, `rank_candidates`, `fetch_venues`, `send_chat`, `show_map`. The system prompt encodes the product rule directly ŌĆö replies capped at two sentences, no tables or lists of numbers in chat, `show_map` mandatory for revealing results.

- The first four tools are thin wrappers around the ADR 0011/0008 analysis funnel, each its own Trigger.dev child task invoked with `triggerAndWait`: `ch-generate-candidates` and `ch-score-candidates` (ClickHouse), and `route-matrix`, which itself fans out dozens of `tfl-journey-plan` children via `batch.triggerAndWait` (ADR 0013). Because the model can call a tool more than once (its own retry, or just asking again), each executor caches its result in an in-memory `AgentState` object and returns the cached value on a repeat call within the run ŌĆö an **at-most-once guard**, since these ClickHouse tasks are not naturally idempotent per invocation.
- `route-matrix` and `ch-score-candidates` separately guard against a retry of the **task itself** (not just a repeated tool call): each starts with `DELETE FROM ... WHERE analysis_id = {id}` before its `INSERT`, because `route_observations` and `candidate_scores` are plain `MergeTree` with no dedup on re-insert ŌĆö a re-attempted task must not double-insert its rows.
- Live UI state rides two Trigger.dev realtime channels, not a Postgres poll: `metadata.set("status"/"map"/"timeline")` for the phase/label and the map overlay (`setStatus`, `execShowMap`, `publishFinalOverlay`), and a `streams.define<string>({id:"agent"})` v2 stream (`packages/tasks/src/trigger/streams.ts`) that the task appends every `response.output_text.delta` chunk to, so the chat panel can render the reply typing live. `apps/web/hooks/use-room-agent.ts` subscribes to both, by run id, via `@trigger.dev/react-hooks`.
- That subscription is authorized by a **run tag**, not a run-id allowlist: `startAnalysis` triggers with `tags: ["room:<roomId>"]`, and `/api/rooms/[roomId]/realtime-token` mints a Trigger public token scoped to `{read: {tags: ["room:<roomId>"]}}` only after `requireMember` passes ŌĆö so a member of room A can never subscribe to room B's runs. This is the same shape as the Liveblocks access-token scoping in [0014](0014-realtime-broadcasts-and-session-identity.md).
- The plan the group ends up seeing is assembled **deterministically** in TypeScript (`assemblePlanResult`) from the cached tool outputs once the loop ends ŌĆö never from the model's own words. The LLM only chooses which tools to call, in what order, and what to say or paint; every number that reaches `plan_snapshots` (scores, minutes, venues) is read back out of `AgentState`, so a hallucinated figure in a `send_chat` call can never become the plan of record. `publishFinalOverlay` likewise recomputes the final map overlay from the winning candidate's true H3 centroid, overwriting whatever the model's own `show_map` call last painted, so the persisted result and the visible map can't disagree.
- `retry: { maxAttempts: 1 }` on `room-agent` itself, overriding the project's default of 3 (`trigger.config.ts`): a Trigger-level retry would re-run the whole tool loop and double-post `send_chat` messages into the room's durable chat history. Failures are instead caught inside the task and turned into a narrated, terminal outcome ŌĆö `failGracefully` posts one apologetic chat message (skipped if the model already narrated the same semantic failure itself), calls `markPlanFailed`, and sets `status:"error"` ŌĆö never rethrown, never retried by the platform.

## Consequences

- New analysis capability is a new tool added to `TOOLS`/`executeTool`, not a new fixed pipeline stage ŌĆö but each addition needs its own `AgentState` cache slot, and its own DELETE-guard if it inserts into ClickHouse.
- The "visual-first, Ōēż2-sentence chat" rule is enforced by the system prompt, not by code ŌĆö `execSendChat` only rejects empty text. Loosening or tightening that product rule is a prompt edit, and nothing stops a future model revision from ignoring it.
- Runs can end in `needs_origins` or a semantic failure (`no_candidates`/`no_routes`/`no_scores`) that are **not** Trigger task errors ŌĆö the run reports success either way. Monitoring "did planning actually work" must read `plan_snapshots.status` / `metadata.status`, not Trigger run status alone.
- `maxAttempts:1` means a transient OpenAI or ClickHouse blip that a retry would have absorbed instead ends the run as failed ŌĆö accepted because re-asking is one click, and the alternative (duplicate chat messages from a replayed run) is worse.
- The `MAX_STEPS` (12) tool-call loop cap and each child task's own `maxDuration` bound the worst case within `room-agent`'s `maxDuration: 900`.
